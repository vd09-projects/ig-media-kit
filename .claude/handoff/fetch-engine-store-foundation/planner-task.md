# Task plan: T1 — Fetch engine + store foundation (anonymous IG reel fetcher)

Task-level breakdown for the foundational plumbing all four MCP tools depend on: an anonymous curl_cffi fetch primitive, a persistent CSV+YAML store with skip-seen dedupe and durable-first partial persistence, load-bearing politeness (a stop-signal classifier plus a **short-circuiting** top-scan page-walk, not just literal 401) baked into the fetch loop, a config schema mirroring yt-media-kit, and a bootable FastMCP skeleton.

**Revision note (round 1 review):** Resolved four mandatory findings — (1) a `stop_signal` classifier replaces literal-401 detection, (2) durable-first / persist-rows-before-cursor ordering with atomic state writes, (3) a precise auth-cookie-keyed definition of ANONYMOUS, and (4) explicit top-scan vs deep-resume traversal modes with disambiguated state roles.

**Revision note (round 2 review):** Resolves one mandatory blocking finding — the `top_scan` stop condition was specified as an ordered comparison (`<= high_water_id`) on Instagram **shortcodes**, which are Base64-ish tokens and are **not lexically orderable**; taken literally the condition never fires, so every top_scan call would page to the ~4-page cap and burn the metered ~48/6.6-min budget — a load-bearing politeness regression that dedupe would silently mask. Fixed by (a) making the **primary** stop condition MEMBERSHIP in the persisted seen-set (stop on the first already-seen shortcode; feed items arrive newest-first, so first-known means caught up), (b) storing the anchor as a **numeric `high_water_media_id`** (from `pk`/`media_id`, not the shortcode) as a monotonic belt-and-suspenders ordered check, and (c) adding an explicit acceptance check in T1.5/T1.6/T1.9 proving the page-walk short-circuits and does **not** always hit the 4-page cap. Round-2 secondary suggestions folded in where they strengthen the plan (noted inline). Accepted in full — see Disposition of round-2 feedback.

## Success Metric
- Primary: Against a real public handle, a single synchronous window call writes store/<handle>.csv with >=1 reel row carrying non-null real play_count and a resolvable video_url, plus store/<handle>.state.yaml carrying user_id, **`high_water_media_id`** (numeric pk of the newest reel ingested — the top_scan anchor), **`seen`** (the per-shortcode dedupe set, source of the primary stop condition), deep_cursor (next_max_id toward scan_depth), and last_stop_reason. A second call in **top-scan mode** with no new posts **short-circuits on the first already-seen shortcode — it must fetch at most 1 feed page and add zero rows** (this is the load-bearing anti-regression check). If new reels exist, the second call adds only those (zero duplicate shortcodes; new reels appear). Verified end-to-end against a live handle before T1 is done.
- Counter-metric (must not regress): Zero authenticated requests on any path — no sessionid/ds_user_id/auth cookies, no login/auth params, ever (benign anonymous cookies permitted, see T1.3). The synchronous path never sleeps and never issues >4 feed pages per call. **A caught-up top_scan must NOT page to the 4-page cap** — the page-walk short-circuits the moment it re-encounters known content; pages spent are proportional to *new* posts, not to the cap. Any throttle/block signal (the whole stop_signal family, not just 401) leaves a valid partial store + saved cursor with no traceback, and no reel that was fetched-and-shown is ever lost to a cursor that advanced past unpersisted rows.
- Evaluation window: one live pilot run + one resume run, within the build session. Evaluator: sindri's verify step / build-session review, against a real handle.

## Assumptions & Grounding
- IG behavior claims (endpoint shapes, play_count location, 12/page cap, cursor format, **whether feed items are returned strictly newest-first**, **that each item carries a numeric `pk`/`media_id` distinct from its `code`/shortcode**, the full set of throttle/block responses, whether anonymous cookies are required) are HYPOTHESES to confirm by probe (T1.2) before the fetch primitive is built on them.
- **Shortcodes are not orderable; media_ids are.** A reel's `code`/shortcode is an opaque Base64-alphabet token — comparisons like `<=` are meaningless on it. The numeric `pk`/`media_id` (also the left half of `next_max_id = {media_id}_{user_id}`) is a monotonically increasing integer and IS orderable. Ordered top_scan logic keys off the media_id; identity/dedupe keys off the shortcode. T1.2 confirms both the newest-first ordering and the presence of a numeric pk.
- yt-media-kit is the ergonomics reference for config schema, filter shape, manifest columns; mirror where it translates.
- "Window" = the paced ~40-item span assembled from up to ~4 feed pages (12/page hard cap) — but a caught-up top_scan window is expected to be far shorter because it short-circuits.
- "Stop signal" = any response indicating IG is throttling, blocking, or challenging us — HTTP 401/403/429, a 302/redirect to a login/challenge URL, or a 200 whose body is a challenge/checkpoint/login-required JSON rather than feed items. Enumerated concretely by T1.2, classified by the T1.3 contract. Distinct from a **"caught-up" stop**, which is a normal, successful end-of-walk (T1.5) — not a throttle.

### T1.3 — HTTP client wrapper + stop_signal classifier + anonymity guard
Thin wrapper over curl_cffi with impersonate="chrome" and the mandatory x-ig-app-id header on every API call. Three load-bearing contracts:
- **(Finding 1) `stop_signal` classifier — the load-bearing contract.** A single function classifies every response into `ok` | `stop(reason)` | `error`, where `stop` covers the whole throttled/blocked/challenged family enumerated by T1.2 (401/403/429, 302-to-login/challenge, 200+challenge-JSON), not just literal 401. The classifier returns a typed reason (e.g. `rate_limited`, `login_redirect`, `challenge`, `forbidden`). Every IG-hitting caller branches on this classifier, never on a bare status-code check. (Note: a `stop_signal` is an *abnormal* stop; the "caught-up" end-of-walk in T1.5 is a *normal* stop and is NOT a stop_signal.) This replaces the v1 "expose 401 distinctly" contract.
- **(Finding 3) ANONYMOUS defined precisely — key off auth cookies, not all cookies.** Anonymous = no `sessionid`/`ds_user_id`/authenticated-session cookies and no login/auth params on any request; benign anonymous cookies (mid/csrftoken/ig_did) ARE permitted and, per T1.2, may be required for the fetch to work. The code-level guard asserts the *absence of auth cookies/params*, not the absence of a cookie jar. If T1.2 shows anonymous cookies are needed, the wrapper keeps a per-process cookie jar that is asserted to never contain auth cookies. This corrects the v1 "no cookie jar" phrasing, which conflated "no authenticated session" with "no cookies at all."
- Distinguish metadata calls (instagram.com, metered) from CDN (fbcdn.net, unmetered, redirect-follow) — carry redirect-follow capability for CDN though CDN download is out of T1 scope. Note: metadata calls must NOT blindly follow a 302, because a login/challenge redirect is a stop_signal — the classifier inspects the redirect target rather than transparently following it.
Depends on: T1.0; gated by T1.2. Done when: the wrapper issues an anonymous GET with the required header; the stop_signal classifier maps each enumerated throttle/block/challenge response to a typed stop reason; and the anonymity guard rejects any request carrying auth cookies/params while permitting benign anonymous cookies.
### T1.4 — Fetch primitive: handle -> user_id
Resolve a public handle to user_id via web_profile_info. Cache resolved user_id into state (avoid re-resolving on resume). A stop_signal on this call returns cleanly (no user_id, typed reason) — it does not raise. Depends on: T1.3, T1.2. Feeds: T1.5, AC2. Done when: a public handle yields a stable user_id, and a stop_signal on resolution returns a clean typed failure.
### T1.5 — Fetch primitive: paced feed pagination + normalization + politeness (AC1 fetch half, AC4)
Paginate /api/v1/feed/user/{id}/ via max_id, cap ~4 pages/call, pace pages ~1-2s. Normalize each item to a reel record — capturing BOTH `shortcode` (the `code`, the dedupe/identity key) AND `media_id` (the numeric `pk`, the ordered anchor) — plus play_count, video_url from video_versions[0].url, fetched_at; filter to product_type=="clips" (product_type dispatch is a switch even now). Politeness (load-bearing, all invariants):
- **(Round-2 finding — the corrected top_scan stop condition, load-bearing.)** The page-walk MUST short-circuit; it must NOT rely on any `<=` comparison of shortcodes (shortcodes are not orderable — such a test never fires and silently pages to the cap). The stop condition is defined as:
  - **PRIMARY — seen-set membership.** The caller supplies the persisted `seen` shortcode set. Walking items newest-first, STOP the page-walk the moment an item whose shortcode is already in `seen` is encountered — first-known means we have caught up to previously-ingested content. This is order-tolerant and is the authoritative stop.
  - **SECONDARY — numeric media_id watermark (belt-and-suspenders).** The caller also supplies the persisted `high_water_media_id` (a numeric pk). Independently stop when an item's `media_id <= high_water_media_id`. Because pk is monotonic and the feed is newest-first (T1.2-confirmed), this catches "caught up" even against an empty/rebuilt `seen` set, and guards a fresh handle. If T1.2 found ordering is not strictly newest-first, this degrades to a per-page scan-for-min but the PRIMARY membership check remains authoritative.
  - Whichever fires first ends the walk. Reaching the end without either firing (genuinely all-new content) or hitting the ~4-page cap are the only other terminations. The "caught-up" stop is a NORMAL success, recorded distinctly from a `stop_signal`.
- **(Finding 1)** Stop and return a partial on the FIRST `stop_signal` of ANY kind (per the T1.3 classifier), not only literal 401. The returned partial carries the typed stop reason so the store can record last_stop_reason.
- NEVER sleep in this synchronous path; never poll during a cooldown (polling extends it).
- **(Finding 4) Two explicit traversal modes.** The primitive takes a `mode` param:
  - `top_scan` — page from the newest item forward and STOP per the corrected stop condition above (seen-set membership, backstopped by the media_id watermark). Surfaces reels posted since the last run. This is the mode T1 wires into the synchronous window call (T1.7), because the Success Metric's "second call adds only new rows" means NEWER posts.
  - `deep_resume` — page older/deeper from the saved deep_cursor (next_max_id) toward scan_depth=90, for backfilling history. Its stop condition is "reached scan_depth or ran out of items or hit the page cap" — NOT the top_scan watermark. The primitive SUPPORTS this mode (same pagination code, different start cursor + stop condition), but T1's synchronous entry point exercises top_scan; a dedicated deep-backfill caller is a follow-up ticket (see Out of Scope). This is the partial disagreement with finding 4 — see the note there.
- Emit, per call: the normalized reel list, the newest-seen `media_id` (candidate `high_water_media_id`) AND its shortcode, the deep_cursor (next_max_id) for the deep path, the count of pages actually fetched (so the caller/reviewer can assert short-circuit behavior), a partial flag, and the typed stop reason — one of `caught_up` (normal), a stop_signal reason (abnormal), `page_cap`, or `end_of_feed`.
Depends on: T1.3, T1.4; gated by T1.2 (needs the pk/shortcode + ordering facts). Done when: a call in top_scan mode against a handle with NO new posts since the anchor returns an empty (or dedupe-absorbed) list, reports `caught_up`, **and reports pages_fetched == 1** (proving the short-circuit — it does NOT reach the 4-page cap); a call in top_scan mode with N new posts returns exactly those and stops at the first known shortcode; a call in deep_resume mode pages from a supplied cursor toward scan_depth; injecting ANY stop_signal (not just 401) mid-run yields a clean partial with the cursor/newest-id intact, a typed reason, and no sleep.
### T1.6 — Store layer: CSV manifest + YAML state, skip-seen, durable-first partial persistence (AC2, AC3)
CSV manifest writer (store/<handle>.csv, token-lean columns mirroring yt-media-kit — including BOTH `shortcode` and `media_id`; TSV fallback if captions carry commas / proper quoting). YAML state (store/<handle>.state.yaml) holding:
- `user_id`.
- **`high_water_media_id`** — the numeric pk of the newest reel ingested; the ordered anchor for top_scan's secondary watermark. Explicitly NOT a "newest shortcode" and never compared with `<=` as a string. (Renamed from v2's `high_water_id` to make the numeric-media_id semantics unambiguous and to prevent a re-introduction of the shortcode-ordering bug.)
- **`seen`** — the persisted per-shortcode dedupe set; the source of top_scan's PRIMARY membership stop condition and of skip-seen dedupe. (May be materialized as an on-disk set derived from the CSV shortcodes rather than duplicated in YAML, provided membership tests are O(1) and the CSV remains the source of truth; T1.6 picks one and states it.)
- **`deep_cursor`** — next_max_id; the anchor for deep_resume backfill.
- **`last_stop_reason`** — including whether the last window ended `caught_up` (normal) vs a stop_signal reason.

State-field roles are disambiguated: `high_water_media_id` + `seen` govern "have we caught up to new posts?" (top_scan); `deep_cursor` governs "how far back have we backfilled?" (deep_resume). They advance independently.
- **(Finding 2) Durable-first ordering + atomic state write — an explicit acceptance check.** The write sequence on every window (including a partial stop) is: (a) flush/append the CSV rows for the items actually normalized and fsync/close so they are durable on disk FIRST; (b) ONLY THEN advance `high_water_media_id` / update `seen` / advance `deep_cursor` to reflect exactly the items that were persisted — never past unpersisted rows; (c) write state.yaml atomically via temp-file + os.replace (atomic rename), never an in-place rewrite. If the process dies between (a) and (c), the worst case is re-fetching already-persisted rows (dedupe absorbs them) — never a skipped-forever reel. This directly upholds "store is never destructively capped."
- Per-shortcode skip-seen dedupe (keyed on shortcode, the stable identity). Never destructively cap — append/accumulate toward scan_depth; top-N computed over the pool later.
Depends on: T1.0; informed by T1.2 (pk/shortcode facts). Consumes: T1.5 output. Done when: a normalized batch writes rows + state; a second write with overlapping shortcodes adds zero duplicates; state round-trips (with `high_water_media_id` stored as a numeric pk and `seen` reflecting all persisted shortcodes); the durable-first ordering is proven — an injected failure after CSV flush but before state write leaves the anchor/cursor NOT advanced past those rows (they re-appear and dedupe on retry), and state.yaml is only ever observed fully-written (temp+rename), never half-written; **AND the anchor round-trips so that a subsequent top_scan short-circuits (feeding T1.5's pages_fetched==1 check) rather than re-walking to the cap.**
### T1.7 — Synchronous window call (wire fetch -> normalize -> store) (AC1 end-to-end, AC3, AC4)
Compose T1.5 (in `top_scan` mode) + T1.6 into one synchronous "fetch a window for handle H" entry point: load state (`user_id`, `high_water_media_id`, `seen`, `deep_cursor`) -> top_scan short-circuiting on seen-membership / media_id watermark -> paced fetch -> normalize -> dedupe -> **CSV rows durable FIRST -> then advance `high_water_media_id` / update `seen` / advance `deep_cursor` for persisted items only -> then atomic state write** -> record last_stop_reason (including `caught_up`). The primitive list_reels and the batch runner both call this (batch adds sleeping later — out of T1 scope). Depends on: T1.5, T1.6. Done when: one call on a live handle produces CSV + state; a re-run in top_scan mode with no new posts **fetches at most 1 page, adds zero rows, and records `caught_up`** (the anti-regression proof); a re-run with new posts adds only reels newer than the anchor and dedupes the rest; ANY stop_signal mid-window still persists a valid partial with rows-before-cursor ordering intact and a typed last_stop_reason.
### T1.8 — FastMCP server skeleton (AC6)
ig_media_kit/mcp_server.py with a FastMCP instance and a __main__ entry so python -m ig_media_kit.mcp_server boots. Register a thin list_reels stub calling T1.7 for one handle to prove wiring — the other three tools are later tickets. Depends on: T1.1, T1.7. Done when: python -m ig_media_kit.mcp_server boots the server and the skeleton tool is registered.
### T1.9 — Live acceptance pass (all ACs)
Run the full flow against a real public handle: confirm AC1-AC6 in one sitting, including the review checks — **a caught-up top_scan short-circuits (pages_fetched == 1, zero new rows, `caught_up` reason) and does NOT page to the 4-page cap** (the round-2 anti-regression check); a non-401 stop_signal is caught and returns a partial; the durable-first ordering holds under an injected mid-write failure; the `high_water_media_id` persisted is numeric and round-trips; only auth cookies are asserted-absent (benign cookies allowed); a second top_scan run with genuinely new posts surfaces exactly those and dedupes the rest. Depends on: all above. Done when: every acceptance criterion + the review checks (round-1 and round-2) are observed live.

## Risks
- **Shortcode-ordering re-introduction (the round-2 bug).** Any future edit that compares shortcodes with `<`/`<=`, or that treats `high_water_media_id` as a shortcode, silently disables the short-circuit and burns the metered budget while dedupe hides the regression. Mitigation: the anchor is named `high_water_media_id` and typed numeric; the primary stop is membership (order-independent); T1.5/T1.7/T1.9 carry an explicit `pages_fetched == 1` acceptance check for a caught-up walk so the regression is caught by test, not in production.
- IG surface drift (endpoints, fields, cursor shape, **the newest-first ordering assumption**, and the stop-signal response set rotate). Mitigation: T1.2 probe gate enumerates the throttle/block family and confirms ordering + pk presence; re-run rune per CLAUDE.md triggers if behavior changed. If ordering stops being strictly newest-first, the membership stop still holds.
- **Stop-signal under-detection.** If a new throttle shape appears that the classifier doesn't recognize, the loop could keep paging and escalate the cooldown. Mitigation: the classifier defaults to `stop(unknown)` for any non-`ok` response it can't positively classify as feed data — fail closed, not open.
- Cooldown escalation under abuse during development. Mitigation: probe sparingly, respect page cap even in T1.2, never poll during cooldown, and the top_scan short-circuit keeps steady-state calls to ~1 page.
- **Partial-write / cursor-skip corruption.** A cursor/anchor advancing past unpersisted rows would silently drop reels. Mitigation: T1.6 durable-first ordering + atomic state rename, with an explicit injected-failure acceptance check.
- **Over-stripping cookies increases blocking.** Removing benign anonymous cookies may raise block rates or break the fetch. Mitigation: T1.2 settles the cookie requirement empirically; the guard keys off auth cookies only.
- **Empty/rebuilt `seen` set.** If `seen` is lost but `high_water_media_id` survives (or vice versa), one anchor must still stop the walk. Mitigation: the two stop conditions are independent and either suffices; a fresh handle with neither anchor legitimately walks to the cap once, then is anchored for subsequent runs.
- Signed-URL expiry mid-test (~36h TTL) could make a stored video_url look broken — not a fetch bug. Mitigation: store fetched_at; judge freshness by it.
- CSV comma collision in captions. Mitigation: TSV fallback rule in T1.6 / proper quoting.
