## 2026-07-16 03:18 — planner task v1 (draft)

Build T4, the async batch subsystem — the *only* background-job path in the IG Media Kit. `start_batch_fetch` must hand back a `job_id` instantly and continue working detached: fill each configured (or requested) handle toward `scan_depth` across escalating IG rate-limit cooldowns (the runner is the system's only sleeper), checkpoint durably after every page so a kill/restart resumes rather than restarts, aggregate a top-N over the full stored pool (cross-channel `global` or `per_channel`), optionally download the top-N mp4s via the T3 downloader, and POST the aggregated result to an arbitrary callback URL with bounded retry+backoff. `get_batch_status` reads job state throughout and returns the final result even if the callback never lands. Task-level breakdown in 9 ordered steps (concurrency overlay active) grounded in the shipped T2/T3 code: a `BatchJob`/`JobPhase` state model + atomic `store/_batch/<job_id>.json` checkpoint reusing `_write_state_atomic` discipline; a serialized per-handle fill loop around `run_window` that sleeps ≥ 1 cooldown window on a metered stop_signal and resumes from the persisted cursor (the only sleep site, never polls during cooldown); a strictly-sequential whole-job driver honoring the single-IP invariant; `global`/`per_channel` aggregation composing `ranking.load_pool/filter_pool/rank/select_top` over the full deduped/watermarked manifests; an optional `download_top` via `run_download_reel`; a callback poster on a bare non-IG transport (no `x-ig-app-id`/cookies) with bounded exponential-backoff retry where result durability is decoupled from callback delivery; and the two entrypoints with detached background launch + a pure-read `get_batch_status`. Concurrency overlay enumerates the shared-state inventory, five under-concurrency invariants, a six-row crash/recovery failure-mode table, an at-least-once callback idempotency stance keyed on `job_id`, and a fault-injection + fake-clock race strategy. Five open questions carried: background execution mechanism (leaning daemon thread), cross-job fetch serialization, callback URL SSRF policy, `batch` config block shape, and checkpoint format (JSON vs YAML).

## 2026-07-16 21:56 — review findings v1 (draft)

Plan review (round 1) of the T4 async batch runner design via multi-perspective-review. Overall **REQUEST CHANGES**. Nine-reviewer backend panel (Reliability/Rate-Limit always-included per project memory). The plan is approve-grade on failure-mode enumeration, checkpoint atomicity, resume idempotency (dedupe + monotonic media_id watermark), and callback anonymity/durability decoupling — but three load-bearing design gaps block build, two rooted in concurrency decisions parked as Open Questions. Blockers: (1) cross-job fetch serialization is deferred to an Open Question, yet two concurrent workers on the single IP is exactly the politeness counter-metric — a process-wide fetch lock/queue must be in the design, not open; (2) kill/restart resume is the PRIMARY success metric but no entrypoint re-attaches an existing job_id after a process restart (daemon thread dies with the server, auto-resume is Out of Scope) — checkpoints are durable but unreachable, so the metric is unmeetable as specified; (3) caller-supplied callback URL is an unaddressed SSRF surface — require https, block private/link-local/metadata IPs, and disable callback redirect-follow (a 3xx could re-enter instagram.com or an internal host). Top suggestions: source the cooldown-sleep duration from WindowOutcome + a persisted escalation counter; have get_batch_status distinguish fetching vs cooldown-sleeping vs dead-worker; one stable result/callback envelope across global/per_channel; add tests for second-start serialization, full-process-restart resume, callback-redirect/SSRF rejection, and unknown job_id. Corroborated (2+ reviewers): the cross-job serialization gap and the resume-entrypoint/dead-worker contradiction.

## 2026-07-16 22:45 — implementation build v1 (draft)
