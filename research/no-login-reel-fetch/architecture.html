The surface

Four MCP tools · two modes

Interactive tools return fast and never block. The batch tool opts into long-running work and reports back through a callback — this is the only place a background job runs.

① list_reelssync · fast
list_reels(handle, count, sort_by, min_views, min_duration, max_age_days, scan_depth=90, fresh_fetch=false)

Returns a sorted manifest of a channel's top reels. One paced window per call; accumulates into the store. No downloads.

② download_reelsync · fast
download_reel(shortcode)

Downloads one reel's mp4 to local disk and returns the path. Re-resolves a fresh signed URL if the stored one has expired.

③ start_batch_fetchasync · job
start_batch_fetch(handles[], top_n, scope="global", sort_by, …filters, download_top=false, callback_url, callback_secret?)

Kicks off a background pull across many channels, aggregates a cross-channel top-N, then POSTs your callback_url. Returns a job_id instantly.

④ get_batch_statusasync · poll
get_batch_status(job_id)

The poll fallback and source of truth: job state, channels done/total, items collected, result path — even if the callback never lands.

Component diagram

One fetch engine, two ways to drive it

The Fetcher — a single paced ~40-item window — is the shared primitive. Interactive tools call it synchronously; the batch runner calls the same primitive in a background job that's allowed to wait out cooldowns.

Solution flows

What each call does

FLOW Alist_reels(handle)synchronous
  1. Top-check — page from the top for new posts until the stored high_water_id; merge.
  2. Deepen — if depth < 90, resume the saved cursor and fetch older items until 90 or the ~40 window budget is spent.
  3. On a rate wall (401) → stop, save cursors, return what's stored (graceful partial).
  4. Filter + sort the pool → return the top count. Call 1 ≈ 40 reels; later spaced calls fill to 90, then cheap refresh.
FLOW Bdownload_reel(shortcode)synchronous
  1. Look up the shortcode → owner handle, stored URL, existing local_mp4.
  2. Already downloaded → return the path.
  3. Else resolve a fresh signed URL (re-scan the owner feed if the stored one expired) → GET → save → update manifest.
FLOW Cstart_batch_fetch(handles[], …, callback_url)async · ~1–2 h for 10 fresh channels
  1. Persist the job, return job_id immediately — the caller is free.
  2. Background: for each handle, run the call-driven fill; on a wall, sleep the cooldown and resume from the checkpoint cursor.
  3. Aggregate all channels → sort → scope="global" top-N across all (or per_channel). Optionally download the top-N.
  4. POST the callback with the result (3× retry). If it never lands, get_batch_status still holds the answer.

Data model

Store layout — CSV for data, YAML for state

Format split on purpose: flat CSV manifests stay token-lean for a future LLM layer; state is a handful of YAML keys. No large JSON blobs.

# the manifest an LLM or human reads — flat, greppable, token-lean
store/<handle>.csv          shortcode, play_count, likes, comments, caption,
                            taken_at, duration, product_type, video_url,
                            local_mp4, fetched_at

# resume state — tiny, human-readable
store/<handle>.state.yaml    user_id · high_water_id · coverage:[{newest,oldest,cursor}] · last_run

# async batch job + its aggregated result
store/_batch/<job_id>.yaml    status · handles · done/total · params · callback state
store/_batch/<job_id>.csv     aggregated top-N manifest across the batch

# downloaded video files
media/<handle>/<shortcode>.mp4

config.yaml                 # channels[], top_reels filters, dirs — mirrors yt-media-kit

Decision trail

What we rejected, and why

The anonymous path caps at ~48 items per window. Every route past that cap was investigated — three by live pilot — before accepting the cap. This is the evidence behind the chosen design.
